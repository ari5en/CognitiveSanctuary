# Cognitive Sanctuary ‚Äî Realignment Refactor Roadmap

> **Authoritative direction document.** All future implementation must follow this.  
> Do NOT revert to task-centric / Pomodoro patterns.

---

## 1. Drift Diagnosis (What Went Wrong & Why)

| Layer | Current (Wrong) Behavior | Expected Behavior |
|---|---|---|
| **Schedule Page** | CRUD task list ‚Äî user creates raw `StudyTask` records independently | Displays `StudySession` blocks generated by `StudyPlanner`; tasks live inside sessions |
| **Sessions Page** | User manually picks tasks, runs a single generic countdown timer | Executes a pre-defined `StudySession`: alternating `FocusSession` ‚Üí `BreakSession` cycles, breaks are explicit and skippable |
| **Burnout evaluation** | Frontend pre-maps mood ‚Üí fixed score (Happy=20) before calling API | Backend `BurnoutCalculator.calculateScore(session, mood)` runs with real `studyDuration` + `breakCount` + mood |
| **Adaptive scheduling** | `recommended_load` stored but never actually changes session structure | After burnout evaluation, `StudyPlanner.adjustSchedule(score)` reshapes the next session's duration and break frequency |
| **Dashboard** | Flat list of sessions + a gauge; no planner hierarchy | Shows today's adaptive session plan, burnout trajectory, and what the planner changed |
| **Planner ‚Üí Session link** | Tasks created first; sessions created ad-hoc when a timer starts | `StudyPlanner` creates `StudySession` records upfront with computed focus/break structure |

## 2. Correct Domain Model & Ownership Hierarchy

```
User (1)
 ‚îî‚îÄ‚îÄ StudyPlanner (1)              ‚Üê one planner per user, owns the schedule
       ‚îî‚îÄ‚îÄ StudySession (many)     ‚Üê planner generates these
             ‚îú‚îÄ‚îÄ FocusSession      ‚Üê child of StudySession; one focus block
             ‚îú‚îÄ‚îÄ BreakSession      ‚Üê child of StudySession; one break block
             ‚îî‚îÄ‚îÄ StudyTask (many)  ‚Üê activities inside this session
                   ‚îî‚îÄ‚îÄ status: Pending | InProgress | Completed

BurnoutRecord (many)               ‚Üê linked to StudySession, written at session end
BurnoutCalculator                  ‚Üê stateless; computes score from session data
```

### Key ownership rules going forward
1. **Tasks cannot exist without a StudySession.** No standalone task creation.
2. **StudySessions are created by StudyPlanner**, not ad-hoc by the Sessions page.
3. **FocusSessions and BreakSessions are generated automatically** from `StudySession.plannedFocusDuration` and `breakIntervalMinutes`.
4. **Burnout score is always computed on the backend** using the real session object ‚Äî the frontend only sends `mood` (integer 1‚Äì4) and `breaksSkipped`.
5. **StudyPlanner.adjustSchedule()** runs after every burnout evaluation and modifies the *next* session's structure ‚Äî the user never adjusts the schedule manually.

## 3. Database Schema Changes

The current schema needs additive changes only ‚Äî no dropping existing tables.

### 3a. `study_sessions` ‚Äî add missing fields

```sql
ALTER TABLE study_sessions
  ADD COLUMN IF NOT EXISTS planned_focus_duration  DOUBLE PRECISION DEFAULT 45,
  ADD COLUMN IF NOT EXISTS break_interval_minutes  INT DEFAULT 45,
  ADD COLUMN IF NOT EXISTS planned_break_duration  DOUBLE PRECISION DEFAULT 10,
  ADD COLUMN IF NOT EXISTS mood                    INT DEFAULT 2,
  ADD COLUMN IF NOT EXISTS breaks_skipped          INT DEFAULT 0,
  ADD COLUMN IF NOT EXISTS status                  VARCHAR(50) DEFAULT 'Planned';
  -- status values: Planned | InProgress | Completed | Cancelled
```

**Rationale:**
- `planned_focus_duration` ‚Äî how long each focus block is (set by planner; adaptive)
- `break_interval_minutes` ‚Äî trigger a break every N minutes (adaptive)
- `planned_break_duration` ‚Äî how long each break should be
- `mood` ‚Äî stored at session end (1=Happy, 2=Neutral, 3=Tired, 4=Exhausted)
- `breaks_skipped` ‚Äî count of breaks user dismissed; used in burnout formula
- `status` ‚Äî session lifecycle state

### 3b. `study_planner` ‚Äî add adaptive state columns

```sql
ALTER TABLE study_planner
  ADD COLUMN IF NOT EXISTS burnout_mode       VARCHAR(50) DEFAULT 'Normal',
  ADD COLUMN IF NOT EXISTS last_updated       TIMESTAMP DEFAULT CURRENT_TIMESTAMP;
  -- burnout_mode values: Normal | Warning | Recovery
```

### 3c. `burnout_records` ‚Äî add mood + breaks_skipped for audit

```sql
ALTER TABLE burnout_records
  ADD COLUMN IF NOT EXISTS mood           INT DEFAULT 2,
  ADD COLUMN IF NOT EXISTS breaks_skipped INT DEFAULT 0;
```

### 3d. `study_tasks` ‚Äî no changes needed
The current structure (`task_id, session_id, title, estimated_time, status`) is sufficient. Tasks remain children of sessions. ‚úÖ

### 3e. Final schema hierarchy (after changes)

## 4. Backend Changes

### 4a. Models ‚Äî what changes

| Model | Change |
|---|---|
| `StudySession` | Add: `plannedFocusDuration`, `breakIntervalMinutes`, `plannedBreakDuration`, `mood`, `breaksSkipped`, `status`. These are persisted fields. |
| `StudyPlanner` | Add: `burnoutMode` field; `adjustSchedule()` must now return a structured object that sets focus/break durations, not just a % load number |
| `BurnoutCalculator` | **No change to the formula** ‚Äî but it must now be called from the backend (not bypassed). The service method `CalculateScore(session, mood)` already exists; it just needs to be wired in |
| `FocusSession` / `BreakSession` | No new DB tables needed. These are used as in-memory OOP constructs during session execution logic. They can remain as-is. |

### 4c. `StudyPlanner` ‚Äî updated adaptive logic

```csharp
// adjustSchedule() must now return structured settings:
public AdaptiveSessionConfig adjustSchedule(double burnoutScore)
{
    if (burnoutScore < 40)        // LOW ‚Äî Normal mode
        return new AdaptiveSessionConfig(focusDuration: 45, breakInterval: 45, breakDuration: 10, mode: "Normal");
    else if (burnoutScore < 75)   // MODERATE ‚Äî Warning mode
        return new AdaptiveSessionConfig(focusDuration: 30, breakInterval: 30, breakDuration: 15, mode: "Warning");
    else                          // HIGH ‚Äî Recovery mode
        return new AdaptiveSessionConfig(focusDuration: 20, breakInterval: 20, breakDuration: 20, mode: "Recovery");
}
```

> **`AdaptiveSessionConfig`** is a new simple record/class with these fields.

## 7. Adaptive Scheduling ‚Äî Correct Behavior

After every session completion, the planner automatically reshapes the **next** session:

| Burnout Score | Mode | Next Session Structure |
|---|---|---|
| 0‚Äì39 | **Normal** | Focus: 45 min, Break: every 45 min, Break duration: 10 min |
| 40‚Äì74 | **Warning** | Focus: 30 min, Break: every 30 min, Break duration: 15 min |
| 75‚Äì100 | **Recovery** | Focus: 20 min, Break: every 20 min, Break duration: 20 min |

This means:
- User **never manually adjusts** focus/break duration
- The planner does it automatically based on burnout history
- The Schedule page **shows** what the planner computed, not what the user set
